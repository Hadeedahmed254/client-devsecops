1. The Old Way (CI Repo Script)

Location:
githubactions/scripts/smoke-test.sh

Mechanism: Triggered by GitHub Actions after a build.

The Flaw: If we manually edited a YAML file in the Manifest Repo, this script would never wake up. It only knew about code changes, not environment changes.

2. The New Way (Manifest Repo / GitOps)

We moved all the "Brain Power" directly into the Manifest Repository (client-devsecops-menifest). Now, the test logic is a permanent resident of the cluster.

‚õìÔ∏è How the Files are Connected (The Chain of Command)

Think of it as a relay race:

Step 1: kustomization.yaml (The Registration)

Role: This is the entry point. It tells ArgoCD: "Hey, keep track of the ConfigMap and the Job."

Connection: It points to the next two files so they are part of every deployment.

Step 2: smoke-test-configmap.yaml (The Manual)

Role: This stores the actual script code (smoke-test.sh) inside a Kubernetes object.

Connection: It doesn't "do" anything on its own. It just waits for the Job to shout: "Hey, I need instructions!"

Step 3: smoke-test-job.yaml (The Engine)

Role: This is the ArgoCD PostSync Hook.

Connection 1 (ArgoCD): The PostSync annotation tells ArgoCD to start this Job only after the pods are updated.

Connection 2 (ConfigMap): The Job mounts the ConfigMap (the Manual) as a volume inside the pod at /scripts/smoke-test.sh.

Execution: The Job starts a container (curl), reads the script from the volume, and runs the test against the bankapp-service.
